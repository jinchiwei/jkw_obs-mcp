## Problem Statement

The daily review (Plan 4) currently synthesizes calendar + vault deltas + autofeeder digests + open tasks into a morning note. The largest unmodeled signal source is email: threads waiting on Jin's reply, FYI from collaborators (PIs, CurieDx, family), meeting prep, deadlines surfaced via inbox. Without it, the review's "Open threads" section can't say "you owe Roberto a reply about the BARDA appointment" because that information lives only in Gmail.

The killer use case for the daily review is *"who is waiting on me?"*. Email is where most of those waits live.

## What Makes This Cool

The 8am review goes from "here's what's on your calendar and what changed in your notes" to "here's what's waiting on you across every input system you actually use." Email is the missing input that makes the review feel like a real chief-of-staff briefing rather than a vault tour.

Gmail's own classification (`category:primary`, `is:important`) does the heavy lifting on noise rejection — Gmail has been training that classifier on Jin's reading patterns for years. The compiler stands on top of that, layers thread-state semantics ("you replied last? they replied last? new activity since the last review?"), and feeds the LLM a curated set rather than a raw inbox dump.

## Constraints

- Mac-only (cluster machines never see email content; matches CalendarAdapter's implicit Mac-only model).
- Read-only OAuth scope (`gmail.readonly`).
- Single-user (Jin's `mrjinch@gmail.com`); multi-account support is out of scope.
- Token cached at `~/.config/jkw-obs-mcp/gmail-token.json` (mode 600), same directory as Anthropic creds.
- Must follow CalendarAdapter's graceful-degrade pattern: any failure (token expired, network, API error, rate limit) returns `[]`, logs, never raises.
- No new pip dependencies are pure additions — `google-api-python-client` and `google-auth-oauthlib` go under a new `gmail` extra in `pyproject.toml`, optional.
- Output `kb/<machine>/email/<date>.md` excluded from git mirror (`obsidian-git` ignore list); iCloud sync OK.

## Premises

1. Gmail's `category:primary -category:promotions` filter is high-precision noise rejection out of the box. We don't need to roll our own newsletter detector. **agreed**
2. The summarization unit is the **thread**, not the message. "Roberto replied in the BARDA thread" is signal; "Roberto replied" without thread context is noise. **agreed**
3. Email summary lives in the vault (`kb/dreamingmachine/email/<date>.md`) for Obsidian discoverability, but is excluded from the cluster git mirror because email is the most-sensitive content jkw-obs-mcp produces. iCloud sync to iPad/iPhone is acceptable (all Jin's account, all locked devices). **agreed**
4. Email compile bundles into `generate_daily_review` rather than running on its own launchd schedule. The 8am scheduled launchd path was abandoned in favor of an on-demand / boot-time trigger model (boot-time trigger deferred to Plan 6+). **agreed**
5. Graceful degradation is mandatory: a Gmail failure shrinks the review's email section to "no recent email" but the rest of the review (calendar, deltas, autofeeder, open tasks) ships unaffected. **agreed**

## Approaches Considered

### Approach A: Sender allowlist
Tight precision: only senders Jin marks as important make it past the filter. Rejected because the list rots and misses one-off important email from new senders (e.g., a new collaborator reaching out the first time).

### Approach B: LLM-classify every message
Pull all inbox, send each to Claude with "important?". Rejected because $0.50–$2/day forever, slow batch runs, still hallucinates importance, and Gmail's existing `IMPORTANT` label already does this with better signal (trained on Jin's actual reads).

### Approach C: Gmail filter + thread-state + LLM summary (CHOSEN)
Three-stage pipeline: (1) Gmail API query with `category:primary -category:promotions newer_than:2d` does noise rejection, (2) thread-state classifier surfaces "waiting on Jin" / "new activity in important thread" / "first message" buckets, (3) Claude renders the curated set into a markdown summary.

## Recommended Approach

**Pipeline:** `GmailAdapter.fetch_recent_threads()` → `_classify_thread_state()` → `EmailCompiler.compile()` → write `kb/<machine>/email/<date>.md` → `DailyReviewGenerator` reads it as a context input.

**Module layout:**

```
src/jkw_obs_mcp/
├── adapter/
│   └── gmail.py                       GmailAdapter — OAuth flow, thread fetch, body extraction
├── compilers/
│   └── email_compiler.py              EmailCompiler — invokes Anthropic, writes kb/email/<date>.md
├── context/
│   └── email_summary.py               load_recent_email_summary() — daily-review reads kb/email/<today>.md
├── generation/prompts/
│   └── email_summary.j2               Jinja template (thread-grouped, role-tagged)
└── generators/
    └── daily_review.py                Modified: reads email_summary, falls back to None on failure

services/launchd/
└── (no new plist — bundled into existing daily-review path)
```

**Auth flow:** `GmailAdapter.__init__()` lazy-loads token from `~/.config/jkw-obs-mcp/gmail-token.json`. Missing token → first call triggers OAuth desktop flow (opens browser, user grants `gmail.readonly`, refresh token cached). Same pattern as autofeeder.

**Thread state classifier (`_classify_thread_state`):**
- `WAITING_ON_YOU` — last message in thread is from someone else, you haven't replied
- `RECENT_ACTIVITY` — thread has new messages since `last_email_run_at`
- `FIRST_TOUCH` — thread is brand new (single message, never seen before)
- `IGNORED` — newsletter/automated/promotional (filtered before this stage)

**Prompt template (`email_summary.j2`):**
- Section 1: "Waiting on you" (highest signal — includes thread subject, last reply, time elapsed, your inferred next action)
- Section 2: "New activity in active threads"
- Section 3: "New conversations" (first-touch, not yet replied)
- Section 4 (optional): "Acknowledgments" (FYI-only, no action expected — kept short)

**Daily review integration:**
- New context loader: `src/jkw_obs_mcp/context/email_summary.py::load_recent_email_summary(vault_root, machine_id) -> str | None`
- Reads today's `kb/<machine>/email/<today>.md` if it exists; returns `None` otherwise
- `DailyReviewGenerator.generate()` calls it during input gathering (parallel to `load_recent_autofeeder_digests`)
- Prompt template gains an `email_summary` section that says "(no recent email summary)" when None
- New section in the daily review note: "Email pulse" — surfaces top 3-5 items from the email summary, calls out anything in `WAITING_ON_YOU` as candidates for "Open threads"

**Bundled run path:** `generate_daily_review` MCP tool now does:
1. (NEW) `EmailCompiler.compile()` — fetch + summarize + write `kb/<machine>/email/<today>.md`. Wrapped in try/except: any error logs and proceeds.
2. (existing) Gather calendar, vault deltas, autofeeder, open tasks, AND the just-written email summary.
3. (existing) Render daily-review template, write `kb/<machine>/daily/<today>.md`.

Email compile is idempotent — running multiple times in a day overwrites the same `kb/email/<today>.md`. The `last_email_run_at` state lives in `~/.config/jkw-obs-mcp/email-state.json` (or co-located in the existing `last-daily-review.json`) and tracks "what's the cutoff for `RECENT_ACTIVITY`."

## Open Questions

1. **Time window for fetch:** start with `newer_than:2d` (last 48h). May need tuning — too narrow misses Friday-night emails on Monday morning; too wide blows past Gmail rate limits and Claude context.
2. **Body extraction:** Gmail returns MIME-multipart messages. Parse `text/plain` part if present; if only `text/html`, strip HTML to text (use `html2text` or roll a regex). Skip raw-HTML body in the prompt — too noisy for the LLM.
3. **Attachments:** mention presence ("[3 attachments: PDF, PDF, PNG]") but don't fetch contents. PDF parsing is Plan 5+ scope creep.
4. **Output retention:** old `kb/<machine>/email/<date>.md` files accumulate. Acceptable initially (each file is small markdown). If vault size becomes a concern, add a 30-day retention policy in a future plan.
5. **Thread permalink:** include `https://mail.google.com/mail/u/0/#inbox/<thread_id>` in summary so Jin can click through to the actual thread? Yes, probably — this is the high-leverage UX detail.
6. **Multi-account:** scoped out for now (single mrjinch@gmail.com). If UCSF email gets added later, second OAuth client and second token file. Plan 5+ scope.
7. **Inferred next action:** the prompt template asks Claude to suggest a one-line "next action" for `WAITING_ON_YOU` threads ("draft 2-line response confirming attendance"). Useful but borderline; if it hallucinates badly, drop it in v2.

## Success Criteria

- `compile_email` MCP tool exists and writes `kb/<machine>/email/<today>.md` with sections (Waiting on you / New activity / New conversations).
- `generate_daily_review` invokes email compile bundled, reads its own output, includes "Email pulse" section in the daily review.
- A Gmail token expiry / API failure / network outage produces an empty email section in the daily review and does not crash the rest of the run.
- "Open threads" in the daily review references at least one email-derived item on the first real run.
- `kb/<machine>/email/` is excluded from the obsidian-git mirror (verified by checking the cluster after a vault sync — file should not appear).
- Test suite gains coverage for: GmailAdapter init/refresh/empty-token, thread-state classifier (4 cases), graceful-degrade paths, daily-review integration with present and missing email summary.
- All tests pass; full suite stays at parity (currently 97).

## Next Steps

Concrete implementation tasks for `/writing-plans` to break down:

1. **GmailAdapter** — OAuth flow (token cache, refresh), `fetch_recent_threads(query, days)` returning structured `EmailThread` objects with messages.
2. **Thread-state classifier** — pure function over `EmailThread` returning `WAITING_ON_YOU | RECENT_ACTIVITY | FIRST_TOUCH | IGNORED`.
3. **Body extraction utility** — `_extract_text_from_message(mime_message) -> str` handling text/plain, text/html, and multipart.
4. **EmailCompiler** — composes adapter + classifier + Anthropic call, writes via `VaultAdapter.write_kb_note(filename, content, subdir="email")`.
5. **Jinja prompt template** `generation/prompts/email_summary.j2` — thread-grouped, sender-tagged.
6. **`load_recent_email_summary` context loader** — reads today's email summary from vault, returns string or None.
7. **`compile_email` MCP tool** — registered in `mcp/server.py`, lazy-builds `EmailCompiler`. Mirrors `compile_raw` lazy pattern.
8. **`generate_daily_review` integration** — bundles email compile call, passes summary into prompt template. Updates `daily_review.j2` with "Email pulse" section.
9. **Tests** — adapter (mocked Google client), classifier (4 thread-state cases), compiler (stubbed Anthropic), graceful-degrade paths, end-to-end with stubbed everything.
10. **Manual smoke test** — real Gmail OAuth bootstrap, real `compile_email` call, verify output, verify daily review now includes email pulse.
