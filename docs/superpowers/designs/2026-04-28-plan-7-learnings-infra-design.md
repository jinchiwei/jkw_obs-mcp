## Problem Statement

The kb at `kb/<machine>/learnings/{constraints,decisions,postmortems}/` works as a convention (5 backfilled notes shipped 2026-04-28), but writing a new note today is a manual prompt to Claude Code. The user has to remember the path, the frontmatter shape, the categories, then chain through `git pull → write → commit → push → reindex` themselves. Friction high enough that proactive logging during sessions doesn't happen.

Plan 7 closes that gap with an MCP tool that bundles the whole cycle behind a single call. Future Claude Code sessions running on any machine — dreamingmachine today, fac/scs/cph/cdx in Plan 8 — can write a learning with one tool invocation and trust it gets to the brain repo, the local index, and every other machine's pull-on-search.

## What Makes This Cool

The kb stops being a passive directory and becomes a live system. Today's flow: "Claude, write me a kb note about X, here's the path, here's the format, also git push, also reindex" → ~10 lines of prompt every time. After Plan 7: "Claude, log this to kb." Done. The friction evaporates, which means the proactive-logging behavior we discussed (Claude announces and writes when something genuinely surprising happens) actually gets used in practice instead of dying to friction.

Bonus: `ensure_brain_repo_fresh` lets cross-machine reads stay automatically up-to-date with a 5-minute cache, without the heaviness of always-on background sync. Fresh enough at the moment of operation, dormant the rest of the time.

## Constraints

- Must work on dreamingmachine (Mac) and Linux clusters (Plan 8). Linux gets the same MCP tool; nothing platform-specific in record_learning itself.
- Must be idempotent on retry. Same call twice with the same content shouldn't double-write or double-commit.
- Must respect existing kb conventions: write-sandbox to `kb/<machine_id>/`, exact category enum, frontmatter format matching the 5 hand-written notes.
- Must NOT block on git push if offline. The local write is the load-bearing operation; sync is best-effort.
- Schema validation at the tool layer. A typo in `category` (e.g., "constraint" vs "constraints") should fail fast at the MCP layer, not silently create a `kb/dreamingmachine/learnings/constraint/` rogue subdir.
- Auto-reindex after write — the user's `search_vault` call in the same session should find the new note. Cost: ~200ms one note's jina-base-zh embedding + sqlite-vec insert.

## Premises

1. **Three categories are enough.** `constraints`, `decisions`, `postmortems`. No new ones added in this plan. Validated as enum at tool layer. **agreed**
2. **Frontmatter is generated by the tool, not by the caller.** Caller passes `title`, `tags`, `applies_to` as structured fields; tool writes the YAML frontmatter. Prevents drift across machines and avoids requiring callers to know the frontmatter schema. **agreed**
3. **Pull before write, push after, retry once on conflict.** Handles the cross-machine concurrent-write race. Offline = local-only write succeeds; user gets `{written: True, pushed: False, reason: "<error>"}` and obsidian-git plugin or manual git push handles propagation later. **agreed (D1)**
4. **Auto-reindex is blocking, not lazy.** ~200ms for one note is invisible; lazy reindex would risk same-session search-misses. **agreed**
5. **No CLAUDE.md in vault root.** User explicitly opted out. Manual invocation only — Claude Code sessions consult the kb when the user asks or when the tool is reached for. **agreed**
6. **`ensure_brain_repo_fresh` caches at 5-minute granularity.** First search_vault in a burst pulls; the next 10 searches in the same session don't repeat. Pull-on-write is uncached (always fresh before commit). **agreed**

## Approaches Considered

### Approach A: Single MCP tool `record_learning` with bundled git ops (CHOSEN)
One tool that does pull → write → commit → push → reindex inline. Caller waits for the whole cycle. Simple mental model: "I called record_learning, the kb has it now."

### Approach B: Two tools, write + sync separated
`write_learning` writes locally + reindexes; `sync_brain_repo` does the git push. Caller responsible for sequencing. More flexibility, more friction. Rejected — the user wants one call.

### Approach C: Background sync daemon
A long-running process watches kb/ for changes and handles git ops async. Heavier, more failure modes, more setup on each machine. Rejected — overkill for the volume (a few notes per week at most).

## Recommended Approach

**Module layout:**

```
src/jkw_obs_mcp/
├── brain_sync/
│   ├── __init__.py
│   └── sync.py                          ensure_brain_repo_fresh + write helpers
├── learnings/
│   ├── __init__.py
│   └── recorder.py                      record_learning core function
└── mcp/server.py                        Modify: register record_learning tool
```

**`record_learning` tool signature:**

```python
@dataclass
class LearningResult:
    written: bool          # True if local file was created (always True on success)
    path: Path             # absolute path to the new file
    pushed: bool           # True if successfully synced to brain repo
    reason: str | None     # if pushed=False, why (offline / auth / etc.)


def record_learning(
    *,
    category: Literal["constraints", "decisions", "postmortems"],
    title: str,
    content: str,
    tags: list[str] = [],
    applies_to: list[str] = [],
) -> LearningResult:
    """Write a kb learning note + sync to brain repo + reindex.

    Path: kb/<machine>/learnings/<category>/<YYYY-MM-DD>-<slug>.md
    Slug: derived from title via kebab-case + truncate to 60 chars
    Frontmatter: title, date, machine, tags, applies_to (all auto-generated)
    """
```

**Schema validation:** category is a `Literal` enum at the type level + a runtime check in the tool dispatch. Title required, must be ≥3 chars (avoid empty/garbage). Content required, ≥50 chars (avoid trivial notes).

**Filename slug rules:**
- Lowercase
- Spaces → hyphens
- Strip non-alphanumeric except hyphens
- Truncate to 60 chars (preserve word boundary if possible)
- Prefix: `<YYYY-MM-DD>-`
- Collision: append `-2`, `-3`, etc. (paranoia — re-run protection)

**Flow:**
```
1. Validate inputs (category enum, title length, content length)
2. Compute path: kb/<machine_id>/learnings/<category>/<date>-<slug>.md
3. ensure_brain_repo_fresh(max_age_minutes=0)  # always pull before write
4. Write file with auto-generated frontmatter + content
5. git add <path>
6. git commit -m "kb: <title>"
7. Try git push. On failure:
   a. git pull --rebase (handles cross-machine concurrent write)
   b. Retry git push once
   c. If still fails: log warning, leave commit local, return pushed=False
8. Reindex (incremental — adds the one new file)
9. Return LearningResult
```

**`ensure_brain_repo_fresh(max_age_minutes: int)` helper:**

```python
def ensure_brain_repo_fresh(*, max_age_minutes: int = 5) -> None:
    """Pull the brain repo if last pull was older than max_age_minutes.

    Cheap no-op when fresh. Logs but doesn't raise on pull failure.
    State file: ~/.config/jkw-obs-mcp/brain-last-pull.json
    """
```

Used by:
- `record_learning` with `max_age_minutes=0` (always fresh before write)
- Optionally `search_vault` and `find_similar` with `max_age_minutes=5` (cached freshness for reads)
- The pull state file (`~/.config/jkw-obs-mcp/brain-last-pull.json`) tracks `last_pull_at` ISO timestamp

**Frontmatter template (auto-generated):**

```yaml
---
title: <title arg>
date: <YYYY-MM-DD>
machine: <machine_id from config>
tags: [<tags arg>]
applies_to: [<applies_to arg>]
---

<content arg>
```

If `tags` or `applies_to` are empty, the field is still emitted as `[]` (consistent shape, easier to grep).

**Error handling:**

| Failure mode | Behavior | Returned status |
|---|---|---|
| Invalid category | Raise immediately, no file written | (raises) |
| Title too short / content too short | Raise immediately | (raises) |
| Pull fails (offline) | Skip, proceed to local write | `pushed=False, reason="pull failed: <err>"` |
| Push fails first attempt | Run `git pull --rebase`, retry push once | (no status update yet) |
| Push fails second attempt | Leave commit local, log warning | `pushed=False, reason="<final error>"` |
| Reindex fails | Log warning, file is still committed | (warning only — `written=True, pushed=True/False`) |

**MCP tool registration in `mcp/server.py`:**

```python
Tool(
    name="record_learning",
    description="Write a kb learning note (constraints / decisions / postmortems) "
    "to kb/<machine>/learnings/<category>/<date>-<slug>.md. Pulls brain repo, "
    "writes file with auto-generated frontmatter, commits, pushes (with retry), "
    "and reindexes. Returns status including whether sync to brain repo succeeded.",
    inputSchema={
        "type": "object",
        "properties": {
            "category": {"type": "string", "enum": ["constraints", "decisions", "postmortems"]},
            "title": {"type": "string", "minLength": 3},
            "content": {"type": "string", "minLength": 50},
            "tags": {"type": "array", "items": {"type": "string"}, "default": []},
            "applies_to": {"type": "array", "items": {"type": "string"}, "default": []},
        },
        "required": ["category", "title", "content"],
    },
),
```

## Open Questions

1. **Brain repo path discovery.** Currently I hardcode `vault_root` from `config.toml`. The brain repo IS the vault (same git directory). So `record_learning` can do `git -C <vault_root> pull/commit/push`. Confirmed compatible with the existing layout.
2. **Should `record_learning` accept an optional `dry_run=True` flag?** Useful for "show me what you'd write without actually writing." Probably out of scope for v1; if Claude wants to confirm with the user before writing, that's a conversation-layer concern, not a tool-layer one.
3. **Auto-detect duplicate writes?** If Claude proposes the same content twice in a session, should we de-dupe? Probably no — cheap to just write again with a `-2` slug suffix and let the user prune later.
4. **Should `ensure_brain_repo_fresh` also push pending local commits?** Edge case: dreamingmachine writes a learning while offline (pushed=False, local commit), then later someone runs `search_vault`. The freshness pull doesn't push pending. Separate `flush_pending_writes` could handle, but probably overkill — obsidian-git plugin already pushes on its schedule.
5. **Reindex granularity.** Plan 2's `Indexer.reindex(scope="incremental")` walks the vault and re-embeds changed files. For a single new file, that's still cheap (~1s for the walk + ~200ms for the one embed). Could optimize with `Indexer.index_one_file(path)` later if it bothers anyone. Not a blocker.

## Success Criteria

- `record_learning` console-tool exists and is invokable from any Claude Code session with the jkw-obs MCP loaded.
- A successful call writes the file at the expected path, with frontmatter matching the 5 hand-written notes' shape, content body verbatim from the call.
- Schema enforcement: invalid `category` raises with a clear error before any I/O.
- Cross-machine race test: two simultaneous `record_learning` calls (one on dreamingmachine, one mocked-as-fac) → both succeed, no file collision (different paths because different machine_ids), pull-rebase handles the divergent push.
- Offline test: `git push` mocked to fail with network error → call returns `pushed=False, reason="..."` but file is written and committed locally; `search_vault` finds it immediately (reindex still ran).
- Auto-reindex confirmed: write a learning, immediately call `search_vault` for its title, hit appears.
- Tests stay green; new tests cover: input validation, slug generation, frontmatter rendering, mocked git ops (pull / commit / push / push-with-conflict / push-offline), reindex hook.
- Manual smoke test: write a real learning via the MCP tool from a Claude Code session, verify on github.com/jinchiwei/jkw_obs-brain that it appears within seconds.
