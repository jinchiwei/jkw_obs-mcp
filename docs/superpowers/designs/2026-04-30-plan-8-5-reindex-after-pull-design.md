## Problem Statement

Plan 7 made the brain cross-machine writable: `record_learning` on machine A pushes to the shared brain repo, and `ensure_brain_repo_fresh` on machine B pulls those changes when `search_vault` is next called. But `search_vault` queries the local sqlite-vec embeddings store, which has no idea about the just-pulled file. Result: machine B's `search_vault` misses content written on machine A until something else triggers a reindex on B.

Surfaced during Plan 8 scs smoke test: scs wrote a learning, the file pushed and pulled to dreamingmachine cleanly, but `search_vault` from dreamingmachine missed it. We had to manually call `reindex` to make it findable.

## What Makes This Cool

The fix is surgical and the architecture stays clean. `ensure_brain_repo_fresh` already knows whether a pull happened. If we make it also report whether HEAD moved, the caller can conditionally reindex — paying the indexing cost exactly once per actual remote change, zero cost when nothing changed. No background daemons, no shared embedding storage, no extra config.

## Constraints

- **No double-reindex on writes.** `record_learning` already calls `indexer.reindex(scope='incremental')` after a local write. The new helper-returns-bool path should only matter for read paths (search_vault, find_similar). record_learning ignores the return value.
- **No extra cost on cache hits.** When `ensure_brain_repo_fresh` short-circuits (cache fresh, no pull attempted), it returns False → no reindex, no overhead.
- **No extra cost on no-op pulls.** When the pull runs but HEAD didn't move (already up to date), returns False → no reindex.
- **Indexer is optional on the adapter.** Some test setups don't wire one. Dispatch must guard `getattr(adapter, "indexer", None) is not None`.

## Premises

1. **`ensure_brain_repo_fresh` returns `bool`** — True iff a pull was performed AND HEAD moved. Cache-hit returns False; pull-no-op returns False; pull-failure returns False. **agreed**
2. **Dispatch branches in `search_vault` and `find_similar` check the return value** and call `adapter.indexer.reindex(scope='incremental')` when True. **agreed**
3. **`record_learning` ignores the helper's return value** — its own reindex covers the local-write case. **agreed**
4. **Pre-pull HEAD is captured via `git rev-parse HEAD`** before the pull. Post-pull HEAD compared against it. Same-SHA → False; different → True. **agreed**

## Approach Considered

Five options were discussed (always-reindex; return-bool-from-helper; background daemon; persist-DB-in-repo; bake-reindex-into-helper). Return-bool-from-helper is the only one that's surgical, has zero cost on the no-change path, and doesn't bleed Indexer dependency into brain_sync.

## Recommended Approach

`ensure_brain_repo_fresh` signature changes from `-> None` to `-> bool`. The helper:
- Returns False if cache hit (no pull attempted)
- Returns False if pull failed (HEAD didn't move)
- Returns False if pull succeeded but HEAD didn't move (already up to date)
- Returns True if pull succeeded AND HEAD moved

Dispatch branches in `mcp/server.py`:
```python
if name == "search_vault":
    pulled_new = ensure_brain_repo_fresh(adapter.vault_root, max_age_minutes=5)
    if pulled_new and getattr(adapter, "indexer", None) is not None:
        adapter.indexer.reindex(scope="incremental")
    # ...existing query code
```

Same shape for `find_similar`.

## Success Criteria

3. No reindex performance regression on the hot path (cache hit → no pull → no reindex). Verified by test that mocks `ensure_brain_repo_fresh` returning False and asserts `indexer.reindex` was never called.
4. `record_learning` continues to reindex once per write (no double reindex).
