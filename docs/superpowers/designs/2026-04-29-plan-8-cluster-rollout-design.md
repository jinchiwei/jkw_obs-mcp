## Problem Statement

jkw_obs-mcp runs on dreamingmachine. Plans 1-7 shipped: vault adapter, semantic search via jina-zh embeddings, compilers, daily review, Gmail, installer, learnings infra. Today, on a remote cluster — scs, fac, cph, teal, cdx — Claude Code sessions have no access to the brain. They can't `search_vault` for "Versa requires VPN", can't `record_learning` when the cluster surprises us, can't `find_similar` to see whether a thought has already been written.

Plan 8 closes that gap on **scs first**, with the bootstrap machinery designed so Plans 9-12 (per-cluster rollouts to fac, cph, teal, cdx) become small mechanical applications of the same machinery. Scope is intentionally minimal: read tools (`read_note`, `list_notes`, `search_vault`, `find_similar`, `reindex`) and `record_learning`. No `compile_raw`, no `compile_email`, no `generate_daily_review` — those stay Mac-only because there's no reason to compile papers or generate calendar reviews on a compute cluster.

## What Makes This Cool

The brain stops being machine-bound. Right now Jin's "second brain" is locked to whatever machine Obsidian + the MCP server are running on — practically, just dreamingmachine. After Plan 8 (scs) and Plans 9-12 (the rest), any Claude Code session on any machine reaches into the same brain. Working at FAC and hit a constraint? `record_learning` from FAC, the fact lands in the brain repo, and the next session on dreamingmachine 30 minutes later already sees it.

The architecture is already mostly there from Plan 7: `record_learning` is machine-id-agnostic by construction (writes to `kb/<machine_id>/learnings/...`), `ensure_brain_repo_fresh` already pulls cross-machine state on demand, the embedder/store already work cross-platform via fastembed ONNX. What's missing is the *bootstrap* — the unattended-install machinery that takes a fresh SSH session into a cluster and turns it into a working MCP node in one command.

## Constraints

- **scs only in this plan.** Other clusters get their own follow-up plans (Plans 9-12) so per-cluster quirks (FAC egress, WSL paths, SageMaker session lifecycle) are captured as kb learnings rather than scope-creep into Plan 8.
- **No compile_raw / compile_email / generate_daily_review on clusters.** Disabled at the MCP tool surface (filtered out in `tools_for_adapter` when `platform.system() != "Darwin"`) and the dispatch branches gated. Compilers need Anthropic creds + Versa VPN + Gmail OAuth that don't make sense on a remote cluster.
- **Single-command bootstrap from a fresh SSH session.** `curl -fsSL https://raw.githubusercontent.com/jinchiwei/jkw_obs-mcp/main/scripts/bootstrap.sh | bash` is the goal. User SSHes into scs, pastes that, walks away, comes back to a working MCP server.
- **Idempotent.** Re-running bootstrap.sh on an already-configured cluster prints status, doesn't reinstall, doesn't break.
- **vault path: `~/arcadia/jkw_obs-brain`.** Bootstrap creates `~/arcadia/` if it doesn't exist (some clusters don't follow Jin's local convention).
- **One embedder model: jina-embeddings-v2-base-zh.** Same model on Mac and Linux. Embeddings are interchangeable across machines.
- **Python 3.11+ via project-local `.venv/`.** uv if available (`uv venv && uv pip install -e .`), stdlib `python3 -m venv` fallback for clusters where uv install is blocked.
- **MCP server registration via `claude mcp add`.** Falls back to print-the-exact-command if `claude` CLI not on PATH yet.
- **Offline-tolerant.** scs has occasional network blips; bootstrap should fail loudly on the affected step but leave prior steps committed (e.g., venv created → clone failed → re-running picks up at clone).

## Premises

1. **scs has normal GitHub egress** (no proxychains required). FAC is the harsh-egress cluster — its quirks are deferred to Plan 9. **agreed**
2. **The 5-cluster rollout is implemented as 5 plans, not 1.** Plan 8 = machinery + scs. Plans 9-12 = apply machinery to fac/cph/teal/cdx, with per-cluster gotchas captured as kb learnings. **agreed (D7)**
3. **Read tools + `record_learning` is the load-bearing scope.** Compile/daily-review explicitly excluded — no value on a compute cluster. **agreed**
4. **The brain repo is ~80MB total** (vault content + embeddings DB). Fits comfortably in any cluster home directory; no need to symlink to scratch. **agreed**
5. **Bootstrap.sh is a thin shell wrapper** (~30 lines: detect Python, create venv via uv-or-stdlib, clone repo, exec the Python installer). The Python installer (`jkw-obs-mcp-setup`) does the heavy lifting (config_dir, machines.toml check, brain repo bootstrap, MCP registration). **agreed (D4)**
6. **Plan 8 ships scs as part of the same plan** — scs is the smoke test for the machinery. We don't ship "machinery alone" because we don't trust untested machinery. **agreed (D7)**
7. **MCP registration uses the official `claude` CLI surface.** No hand-written `~/.claude.json` — Anthropic owns that schema, not us. **agreed (D6)**

## Approaches Considered

### Approach A: bootstrap.sh + extended Python installer (RECOMMENDED)

Single-command bootstrap from a fresh SSH session.

**`scripts/bootstrap.sh`** (~30 lines): a public, curl-able shell script that:
1. Checks for Python 3.11+ (errors out with a useful message if missing).
2. Ensures `~/arcadia/` exists (`mkdir -p`).
3. Clones `jkw_obs-mcp` to `~/arcadia/jkw_obs-mcp` if not already there.
4. Creates `.venv/` at `~/arcadia/jkw_obs-mcp/.venv` via `uv venv` if `uv` is on PATH, else `python3 -m venv .venv`.
5. Activates the venv and runs `uv pip install -e .` (or `pip install -e .` on the fallback path).
6. Execs `jkw-obs-mcp-setup` (the existing Python installer, extended below).

**Python installer extension** (`src/jkw_obs_mcp/installer/`):
- `bootstrap_brain_repo.py` — clones `jkw_obs-brain` to `~/arcadia/jkw_obs-brain` if not present; runs `git pull --ff-only` if it is; writes `~/.config/jkw-obs-mcp/config.toml` with `vault_root = "~/arcadia/jkw_obs-brain"` and machine_id auto-detected.
- `mcp_registration.py` — runs `claude mcp add jkw-obs --command jkw-obs-mcp` if `claude` is on PATH; else prints the exact command to copy-paste with a clear "you'll need to run this after installing Claude Code" message.
- `cli.py` extended to call these two new steps after the existing config_dir + machines_check steps. Steps 5 (`bootstrap_brain_repo`) and 6 (`mcp_registration`) run on all platforms (no Darwin gate).
- Linux-specific tools_for_adapter filtering: when `platform.system() != "Darwin"`, exclude `compile_raw`, `compile_email`, `generate_daily_review` from the tool surface so Claude Code on the cluster doesn't see them at all.

**Reuses:**
- Existing `installer/cli.py` orchestrator pattern (Steps 1-4 already exist).
- Existing `installer/config_dir.py` and `installer/machines_check.py` modules.
- Existing `record_learning`, `search_vault`, `find_similar`, `read_note`, `list_notes`, `reindex` MCP tools — no per-tool changes needed.
- Existing machines.toml entries for scs/fac/cph/teal/cdx — already in place.

**Effort:** M (human: ~3 days / CC: ~2 hours)
**Risk:** Low — most code paths exist; the new bits are bootstrap.sh, two installer modules, the platform filter in tools_for_adapter.
**Pros:**
- One command from fresh SSH to working MCP.
- Per-cluster quirks captured as kb learnings during scs rollout, ready to inform Plans 9-12.
- Minimal new code surface, mostly extension of existing patterns.
**Cons:**
- bootstrap.sh and Python installer are two surfaces to keep aligned.
- Plan 8 spans both Python (installer modules) and shell (bootstrap.sh) — slightly more complex than a pure-Python plan.

### Approach B: Pure-Python installer, manual clone+venv before bootstrap

User manually clones the repo and creates a venv, *then* runs `jkw-obs-mcp-setup`. The Python installer does everything from there.

**Effort:** S (human: ~1 day / CC: ~1 hour)
**Risk:** Low
**Pros:**
- One file to maintain.
- No shell script.
**Cons:**
- Three manual steps before "one command" — clone, venv, install — which adds friction every cluster. By cluster #4 you'll have made the same mistake twice and spent more time fixing it than the shell script would have cost.

### Approach C: Per-cluster bespoke scripts

Each cluster gets its own custom bootstrap (e.g., `scripts/bootstrap-scs.sh`, `scripts/bootstrap-fac.sh`).

**Effort:** L (human: ~1.5 weeks / CC: ~6 hours)
**Risk:** Medium
**Pros:**
- Maximum flexibility per cluster.
**Cons:**
- 5 scripts to maintain.
- Defeats the "uniform brain" mental model — cluster differences leak into bootstrap rather than being captured cleanly as kb learnings.
- We don't actually know what's different per cluster yet — premature.

## Recommended Approach

**Approach A.** The shell-then-Python split is the right shape for this domain: shell handles "I have nothing", Python handles "I have a tested package and want to do per-step idempotent logic." The two surfaces stay aligned because bootstrap.sh's job is narrow (get to `pip install -e . && jkw-obs-mcp-setup`) and the Python installer's job is the substantive work.

The platform-filter in `tools_for_adapter` is a small, explicit gate. Linux clusters see exactly the 6 tools that make sense there: `read_note`, `list_notes`, `search_vault`, `find_similar`, `reindex`, `record_learning`. Mac sees all 10.

## Open Questions

1. **Which Python ships on scs by default?** If it's already 3.11+, no work. If it's 3.9, bootstrap.sh needs a clear "install Python 3.11+ first" error. Plan 8 Task 1 (manual recon) confirms before we commit. If pyenv/conda/system-Python-managed-by-modules is the answer, we adapt — but don't hard-code module-system assumptions into bootstrap.sh.
2. **Is `uv` already on PATH on scs?** If yes, the install path is fast. If no, the stdlib fallback works fine — slower but bulletproof. Plan 8 doesn't need to resolve this; the fallback handles it.
3. **MCP server lifecycle on the cluster.** On Mac, launchd starts jkw-obs-mcp at boot. On scs, there's no launchd — and we explicitly opted out of compile/daily-review which is what made an always-on server matter. For the cluster, the MCP server runs on-demand via `claude mcp` (Claude Code spawns it per session). No daemon, no systemd unit. Confirm via Plan 8 smoke test.
4. **Does Claude Code work on scs?** scs has Claude Code installed via the FAC tunnel (Jin's reverse-SOCKS setup). Bootstrap.sh assumes Claude Code is or will be installed; the "print-the-command" fallback handles "Claude Code not installed yet" gracefully.
5. **Bootstrap.sh authentication for clone.** `git clone https://github.com/jinchiwei/jkw_obs-mcp.git` works for the public mcp repo. `git clone git@github.com:jinchiwei/jkw_obs-brain.git` requires SSH key auth on the cluster. Plan 8 Task to confirm the brain repo is reachable from scs with Jin's existing GitHub SSH key (or document the auth setup as a precondition kb learning).

## Success Criteria

1. From a fresh SSH session into scs, `curl -fsSL <bootstrap-url> | bash` produces a working MCP node in <5 minutes.
2. A new Claude Code session on scs sees 6 jkw-obs tools (not 10 — Mac-only ones filtered out).
3. `search_vault` from scs returns the same top hits as `search_vault` on dreamingmachine for the same query (modulo tied scores).
4. `record_learning` from scs writes to `kb/scs/learnings/...`, commits, pushes, and the file appears on github.com/jinchiwei/jkw_obs-brain within seconds. Within 5 minutes (cache window), `search_vault` from dreamingmachine finds the new note.
5. Re-running bootstrap.sh on the configured scs is idempotent: prints status, exits cleanly, no breakage.
6. Tag `plan-8-complete` shipped to origin once smoke test passes.

## Distribution Plan

`bootstrap.sh` lives at `scripts/bootstrap.sh` in the public `jkw_obs-mcp` repo. User runs:

```bash
curl -fsSL https://raw.githubusercontent.com/jinchiwei/jkw_obs-mcp/main/scripts/bootstrap.sh | bash
```

No package manager publish, no container image. The shell script + the existing `pip install -e .` from a clone is the distribution.
