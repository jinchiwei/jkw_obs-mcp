## Problem Statement

Plan 4 shipped a launchd plist template referencing a console-script `jkw-obs-mcp-daily-review` that doesn't exist. The boot trigger can't fire until that gap is closed. Separately, every machine setup (dreamingmachine today, scs/fac/cph/teal/cdx eventually) re-does the same manual steps: create config dir, scaffold `.env`, register hostname in `machines.toml`, `pip install -e`, set up secrets. Documenting these in `README.md` doesn't prevent forgetting a step at 2am on a new cluster.

The deliverable: a single command, `jkw-obs-mcp setup`, that works on both Mac and Linux, runs the shared steps unconditionally, runs platform-specific steps (launchd plist, Gmail OAuth) only when appropriate, and leaves the machine in a usable state.

## What Makes This Cool

The same script onboards dreamingmachine, fac, scs, cph, teal, and cdx with one command each, even though Mac and Linux have very different installation surfaces. The platform detection is honest: on Linux, the script never pretends launchd works; it just skips it cleanly and tells you what got configured vs. skipped. On Mac, the script also closes the actual functional gap left by Plan 4 — daily review now genuinely fires on boot when today's review hasn't run yet.

The boot trigger itself is dead simple: read one JSON state file, compare today's date to last_run_at's date, run if different, exit 0 otherwise. Cost per boot: ~10ms. Cost when it fires: identical to running `generate_daily_review` from the MCP tool surface.

## Constraints

- Must work on Mac (Darwin) and Linux without separate codepaths the user has to remember.
- Cluster machines (FAC HPC, etc.) may run Python in a conda env that's NOT `deepdream` — the setup script must detect Python interpreter or accept an override.
- Token files (Gmail, .env) must remain mode 600. Setup must never widen perms.
- Idempotent: running `jkw-obs-mcp setup` twice must not blow anything up. Re-running on an already-configured machine should be a no-op (or "everything's fine, here's status").
- The boot-trigger script must NEVER block on first boot if config is incomplete — it should detect missing config and exit 0 silently.
- Must respect existing manual setup. If `~/.config/jkw-obs-mcp/.env` already exists, don't overwrite it; offer to merge or skip.

## Premises

1. **Platform detection via `platform.system()` is sufficient.** Mac = `Darwin`, Linux = `Linux`. We don't need finer-grained detection (no need to distinguish Ubuntu vs RHEL; both work the same for our purposes). **agreed**
2. **Daily-review boot trigger is Mac-only by design.** Linux clusters don't run an interactive desktop session; "boot trigger" semantics don't apply. Cluster daily-review (if wanted) gets a separate trigger mechanism in Plan 7. **agreed**
3. **Setup is for personal use, not packaged distribution.** No homebrew formula, no `.deb`, no installer pkg. Just `git clone + cd + jkw-obs-mcp setup`. **agreed**
4. **Existing config must be preserved.** Re-running setup on dreamingmachine (which is already fully configured) should detect that and report status, not clobber. **agreed**
5. **Boot trigger uses date-not-time comparison.** "Has today's review already run?" not "have N hours passed since last run?" Sleep across midnight = next boot triggers. Multiple boots same day = one run total. **agreed**

## Approaches Considered

### Approach A: Just close Plan 4's gap (rejected)
A console-script + plist update + tiny `install.sh`. ~80 lines. Doesn't solve cluster onboarding. Rejected because Jin explicitly wants seamless cluster install.

### Approach B: Platform-aware `jkw-obs-mcp setup` (CHOSEN)
A unified `setup` command in `pyproject.toml` `[project.scripts]` that detects platform and runs the appropriate steps. Boot trigger is one of those steps (Mac-only). Closes Plan 4's gap and pre-builds the cluster onboarding story.

### Approach C: Full lifecycle (rejected)
B plus `--update` and `--uninstall` modes. Over-engineered for a single-user single-version tool. Reconsider when there's a second version with different LaunchAgent paths.

## Recommended Approach

**Module layout:**

```
src/jkw_obs_mcp/
├── installer/
│   ├── __init__.py
│   ├── cli.py                          `jkw-obs-mcp setup` entry point
│   ├── config_dir.py                   create ~/.config/jkw-obs-mcp/, scaffold .env
│   ├── machines_check.py               validate/add hostname in machines.toml
│   ├── launchd.py                      Mac: install/uninstall LaunchAgent (skipped on Linux)
│   └── gmail_oauth.py                  Mac: walk through Google Cloud + bootstrap token
└── triggers/
    ├── __init__.py
    └── daily_review_runner.py          `jkw-obs-mcp-daily-review` boot-trigger console-script

services/launchd/
└── com.jinchiwei.jkw-obs-mcp.daily-review.plist   Modified: RunAtLoad=true, no StartCalendarInterval
```

**`jkw-obs-mcp setup` flow** (single command, platform-aware):

1. **Detect platform.** Print `Detected: Darwin` or `Detected: Linux`.
2. **Detect Python interpreter.** `sys.executable` — used to populate the LaunchAgent's `ProgramArguments` so it points at the right env regardless of which conda env the user is in.
3. **Config dir.** `mkdir -p ~/.config/jkw-obs-mcp/`. If `.env` doesn't exist, copy from repo's `.env.example` and `chmod 600`. Print "scaffolded" or "already exists, skipping."
4. **Hostname in machines.toml.** Read current hostname. Check if it's in the registry. If not, prompt user to add it (offer to auto-append with `os = "darwin"|"linux"` as default).
5. **Pip install.** Skip — assume user already ran `pip install -e ".[gmail,mac]"` or equivalent before invoking setup. Print a check that the package is importable; abort with clear message if not.
6. **Gmail OAuth (Mac only).** If `darwin` and no `gmail-token.json`, walk user through Google Cloud setup with copy-pasteable instructions, prompt them to drop `client_secret.json` at the right path, then trigger first OAuth flow. Skipped on Linux silently (with a one-liner: "Gmail compile is dreamingmachine-only; skipping on $HOSTNAME").
7. **LaunchAgent install (Mac only).** Render the plist template with the actual `sys.executable` path and `Label`. Copy to `~/Library/LaunchAgents/`. Run `launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/com.jinchiwei.jkw-obs-mcp.daily-review.plist`. Verify with `launchctl print gui/$(id -u)/com.jinchiwei.jkw-obs-mcp.daily-review`. Skipped on Linux.
8. **Final status report.** Print a summary table: ✓/skipped per step. Tell user what to do next (e.g., "Restart Claude Code to pick up the MCP server").

**Boot-trigger console-script (`jkw-obs-mcp-daily-review`):**

```python
def main() -> int:
    """Entry point invoked by launchd at user login (RunAtLoad=true).

    Reads ~/.config/jkw-obs-mcp/last-daily-review.json. If today's date is
    later than last_run_at's date, runs generate_daily_review. Otherwise
    exits 0 silently. Total cost when no-op: ~10ms.
    """
    config_path = Path.home() / ".config" / "jkw-obs-mcp"
    state_path = config_path / "last-daily-review.json"

    if not state_path.exists():
        # Fresh machine, no review ever run — fire it.
        return _run_daily_review()

    try:
        data = json.loads(state_path.read_text())
        last_run = dt.datetime.fromisoformat(data["last_run_at"])
    except (json.JSONDecodeError, KeyError, ValueError):
        # Corrupt state — treat as never-run.
        return _run_daily_review()

    today = dt.date.today()
    if last_run.date() < today:
        return _run_daily_review()

    # Already ran today — silent no-op.
    return 0


def _run_daily_review() -> int:
    """Build the adapter + generator from config and call generate()."""
    # Same wiring as src/jkw_obs_mcp/mcp/server.py:main(), but synchronous
    # and exits after one generate(). Errors are logged but never re-raised
    # (LaunchAgent doesn't need a stack trace).
    try:
        from jkw_obs_mcp.config import load_config
        # ... (see Plan 6 implementation plan for full code)
        return 0
    except Exception as exc:
        # Log to stderr (LaunchAgent captures to ~/Library/Logs/...)
        print(f"daily-review trigger failed: {exc}", file=sys.stderr)
        return 1
```

**Updated plist** (replaces Plan 4 template):
- Drop `StartCalendarInterval` (the 8am model)
- Add `StartInterval=300` (fires every 5 min while awake; paused during sleep)
- Add `RunAtLoad=true` (also fires at session login as a backup)
- Keep `StandardOutPath` / `StandardErrorPath` for ~/Library/Logs/

**Why `StartInterval=300` over `RunAtLoad` alone:** Jin almost always wakes the laptop from sleep rather than fully shutting down, so `RunAtLoad` (which fires only at session login, NOT on wake-from-sleep) would rarely fire under his usage pattern. `StartInterval=300` means launchd fires the trigger every 5 min while the laptop is awake; on wake-from-sleep, the timer resumes and the next fire happens within 5 min. The boot-trigger script's date check is ~10ms, so the steady-state cost is 12 fires/hour × 10ms = 120ms/hour, which is invisible. After the first fire of the day, every subsequent check is a no-op (date matches). `RunAtLoad=true` is kept as a backup in case the user does a full restart — fires immediately at login regardless of `StartInterval` cadence.

**`pyproject.toml` additions** under `[project.scripts]`:

## Open Questions

1. **Conda env detection.** On dreamingmachine, the MCP server runs in `deepdream`. The installer should record `sys.executable` at setup time so the plist points at the right interpreter. But what if Jin moves the conda env? Probably out of scope — re-run `jkw-obs-mcp-setup` to re-register.
2. **What's the cluster daily-review trigger?** Plan 7 territory. Options: (a) cron, (b) systemd timer, (c) manual, (d) skip entirely. Likely (d) for now since clusters don't have calendar/email/personal context that makes a daily review valuable.
3. **Should setup also `pip install -e`?** Currently I have it skip and assume user did it. But the cleaner UX is `jkw-obs-mcp-setup --bootstrap` that does the pip install first. Defer — adds complexity for marginal gain.
4. **What happens if hostname changes?** (e.g., Jin's laptop hostname rotates between docking states.) `machines.toml` has `hostname_aliases`; setup could append on a mismatch. Defer to actual problem.
5. **Mac sleep-wake behavior.** RESOLVED: switched to `StartInterval=300` (every 5 min while awake) which handles the wake-from-sleep case — Jin's actual usage pattern. `RunAtLoad=true` is also set as a belt-and-suspenders backup for full restarts. Worst case: daily review fires within 5 min of waking the laptop. Steady-state overhead: 120ms/hour, invisible.

## Success Criteria

- `jkw-obs-mcp-setup` exists as a console-script and is invokable from any directory.
- Running it on dreamingmachine (already configured) prints a clean status report — no errors, no overwrites.
- Running it on a fresh Mac (clean slate, just cloned the repo) gets to "MCP server works + LaunchAgent registered + Gmail OAuth bootstrapped" with no other manual steps beyond filling in `.env` values and downloading the Google client_secret.json.
- Running it on a fresh Linux box (e.g., a clean Ubuntu VM mimicking a cluster) gets to "MCP server works + hostname in machines.toml" with Gmail/launchd/calendar steps cleanly skipped (with explanatory messages).
- `jkw-obs-mcp-daily-review` (the boot-trigger console-script) is invokable from launchd, no-ops if today's review already exists, fires `generate_daily_review` if not, logs to ~/Library/Logs/.
- After `jkw-obs-mcp-setup` runs on dreamingmachine, restarting the laptop (or logging out + back in) results in `kb/dreamingmachine/daily/<today>.md` existing within ~1 minute.
- Test suite stays green. New tests cover: platform detection, idempotency of setup, boot-trigger date-comparison logic, plist rendering with real `sys.executable`.
